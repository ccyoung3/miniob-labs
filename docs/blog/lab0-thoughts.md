# Lab 0 学习记录

> 在完成 MiniOB 的 Lab 0 布隆过滤器（Bloom Filter）组件时，记录下了对于并发控制和性能瓶颈的客观分析。

## 0. 背景说明

在初步接触 Lab 0 的题目时，核心目标是使用 Hash 映射 Bit 数组。
进一步分析实验要求后，确认需要在多线程并发插入和查询场景下保证线程安全和高性能。对于常规实现方式，通常会在 `put` 和 `contains` 方法前使用 `std::shared_mutex`（读写锁）。

然而，在运行多线程高并发压测时，发现了如下问题：频繁获取读写锁导致了严重的线程上下文切换开销，CPU 占用率高，但吞吐量明显受限。
这说明作为数据库引擎底层的高频组件，锁的代价较高，需要考虑无锁化（Lock-Free）实现。

## 1. 锁与 `std::atomic` 的应用

布隆过滤器的本质是 Bit 数组。在插入数据时，只需将 Hash 计算出的几个固定位置的 Bit 置为 1。由于置 1 操作是幂等的，因此无需为局部冲突对整体加锁。

在具体实现中，采用了 C++11 提供的原子操作。将底层的 Bit 数组声明为 `std::atomic<uint64_t>` 的数组。当需要将第 $n$ 个 bit 置 1 时，定位到对应的 `uint64_t` 并使用原子操作：
```cpp
bits_[index].fetch_or(mask, std::memory_order_relaxed);
```
通过原子的按位或 (`fetch_or`) 操作，消除了锁的开销，提升了并发插入速度。查询时作为纯读操作，使用 `memory_order_relaxed` 也能以较低的代价获取最新状态。

## 2. 哈希函数的选择与假阳性 (False Positive) 问题

在实现过程中，初期尝试了简单的乘法哈希。测试结果显示了较高的 False Positive（假阳性）率，主要因为哈希分布不均匀导致不同元素映射冲突增加。

随后换用了 `MurmurHash3` 系列算法。为计算 $K$ 个哈希值，采用了通过两个基础哈希值进行线性组合的双重哈希策略：
$Hash_i(x) = (HashA(x) + i \times HashB(x)) \pmod m$
此方法计算开销小，且分布均匀。

此外，在使用 Hash 值作为索引前，需要确保对 Bit 数组的总长度取模，以避免数组越界导致的段错误 (Segmentation Fault)。

## 3. 进阶考量：伪共享 (False Sharing) 的隐患

复盘时关注到了一个更底层的性能隐患。虽然使用了原子操作去除锁控，但现代 CPU 以 Cache Line（通常为 64 字节）为单位加载内存。
如果两个线程高频并发修改相邻位置的 Bit（同属一个 Cache Line），即使修改不同的 `uint64_t`，也会触发 CPU 缓存一致性协议（如 MESI），导致缓存行失效（Cache Invalidation）。此现象称为**伪共享 (False Sharing)**。

针对此问题，虽然在当前的实验规模中可能不构成主要瓶颈，但在工业级数据库中通常需要对齐 Cache Line 或使用分片 (Sharding) 技术来分散热点。

完成 Lab 0 后，对底层硬件特性和并发原语有了进一步认识，为后续实验打下了基础。

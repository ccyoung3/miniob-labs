# Lab0 基础组件踩坑笔记：并发布隆过滤器的无锁之美

> 声明：本笔记为《当代数据管理》大作业的个人学习记录。在完成 MiniOB 的 Lab 0 布隆过滤器（Bloom Filter）组件时，记录下了对于并发控制和性能瓶颈的一些思考。

## 0. 写在前面的话

在刚拿到 Lab 0 的题目时，我的第一反应是：“不就是一个用 Hash 映射 Bit 数组的结构吗？这有何难？”
但细看实验要求后，发现需要在多线程并发插入和查询场景下保证线程安全和高性能。对于平时习惯写业务代码的我来说，遇到并发问题，第一反应往往是简单粗暴地加一把大锁：直接在 `put` 和 `contains` 方法前上一把 `std::shared_mutex`（读写锁）。

然而，当我跑起多线程高并发压测时，悲剧发生了：原本应该极度轻量级的布隆过滤器，反而因为锁竞争导致了严重的线程上下文切换开销，CPU 占用率飙升，但吞吐量却少得可怜。
这让我意识到，作为数据库引擎底层的高频组件，**锁的代价是极其高昂的**，必须想办法实现无锁化（Lock-Free）。

## 1. 锁的幻觉与 `std::atomic` 的魅力

为什么布隆过滤器天然适合无锁操作？
仔细剖析布隆过滤器的本质，它其实就是一个巨大的 Bit 数组。在插入数据时，我们只需要将 Hash 算出来的几个固定位置的 Bit 置为 1 即可。由于置 1 操作是幂等的（不管几个线程同时把这个位置 1，最终结果都是 1），这意味着我们完全没有必要因为某些位发生冲突而把整个数组锁起来。

于是，我摒弃了 `std::mutex`，转而求助于 C++11 提供的原子操作。
在实现中，我将底层的 Bit 数组声明为 `std::atomic<uint64_t>` 的数组。当需要将第 $n$ 个 bit 置 1 时，只需要定位到对应的 `uint64_t`，然后使用原子操作：
```cpp
bits_[index].fetch_or(mask, std::memory_order_relaxed);
```
通过原子的按位或 (`fetch_or`) 操作，我们彻底摆脱了锁的开销，极大地提升了并发插入速度。查询时由于是纯读操作，使用 `memory_order_relaxed` 也能以极低的代价拿到最新状态。

## 2. 哈希函数的选择与假阳性 (False Positive) 踩坑

在这个过程中，我还踩了哈希函数的坑。起初为了简单，我用了自己随便搓的一个乘法哈希。结果测试用例一跑，大量的 False Positive（假阳性，即判断存在但实际不存在）让我傻眼了。因为哈希分布不均匀，大量的不同元素被映射到了同一个位上。

最终我换用了业界经典的 `MurmurHash3` 系列算法。由于我们要计算 $K$ 个哈希值，没必要真的调用 $K$ 次复杂的哈希函数，我采用了通过两个基础哈希值进行线性组合的方法（双重哈希策略）：
$Hash_i(x) = (HashA(x) + i \times HashB(x)) \pmod m$
这样的计算开销极小，而且分布非常均匀。

另外还有一个新手很容易犯的错误：**数组越界导致段错误 (Segmentation Fault)**。在使用 Hash 值作为索引之前，一定不能忘记对 Bit 数组的总长度取模！

## 3. 进阶思考：伪共享 (False Sharing) 的隐患

在最终复盘这个 Lab 时，我想到了一个更底层的性能隐患。虽然我们用了原子操作去除了锁，但是现代 CPU 是以 Cache Line（通常是 64 字节）为单位加载内存的。
如果两个线程高频并发地 `fetch_or` 修改相邻位置的 Bit（它们恰好位于同一个 Cache Line 中），即使它们修改的是不同的 `uint64_t`，也会触发 CPU 缓存一致性协议（如 MESI），导致缓存行失效（Cache Invalidation），这就是经典的**伪共享 (False Sharing)** 问题。

针对这一点，虽然在当前的 MiniOB 实验规模中可能不构成主要瓶颈，但在工业级数据库（如 RocksDB 的过滤器实现中）往往需要对齐 Cache Line 或使用分片 (Sharding) 技术来打散热点。

完成 Lab 0 后，我深刻感受到了写底层数据库代码的乐趣所在——在这里，每一行代码的性能开销都会被放大，唯有对底层硬件和并发原语有敬畏之心，才能写出真正高性能的引擎。接下来就是更大的挑战了——Lab 1 LSM-Tree！

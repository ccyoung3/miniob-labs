# Lab 2: 查询引擎 - 优化器与 Hash Join 的踩坑与心得

在 Lab 2 的实验中，我的主要目标是打通 SQL 的查询引擎。从修改 `lex`/`yacc` 解析语法树开始，到在查询优化器中增加等值谓词的智能改写，最后实现火山模型下的 `HashJoinPhysicalOperator` 物理算子。在这个过程中，我在 C++ 的指针传递、生命周期管理以及数据库内核组件的解耦设计上吃了不少苦头，也收获了非常多。

以下是实现过程中的核心感悟与踩坑记录。

---

## 💡 踩坑笔记 (Gotchas)

### 1. 悬挂指针 (Dangling Pointers) 与元组拷贝
**现象**：在使用 `HashJoin` 时，需要在内存中将左表扫描出来的 Tuple 放进哈希桶 (`std::unordered_map`) 里。起初我直接保存了 `left_tuple` 的指针。然而当在调试时，却发现匹配到的左表数据都是相同的值或者出现了段错误 (Segfault)。
**原因**：数据库执行引擎为了节省内存申请的开销，通常在每次 `next()` 调用时复用底层的 Buffer，导致 `left_tuple` 所指向的内存区不断被新拉取的数据覆盖。
**解决**：不能存原指针，而是必须进行**深拷贝**。本项目中我们借助于 `ValueListTuple::make` 工具方法，将每一行里的具体字段值 (Value) 全部独立克隆出来，存储到内存桶中。数据库这种流式数据处理中，明确对象的生命周期极其重要。

### 2. 表达式求值时的 Tuple 上下文
**现象**：在匹配 `INNER JOIN ... ON a.id = b.id` 这一条件时，等号左右两侧分别属于左表和右表。但是 `Expression::evaluate(tuple)` 接口只接受一个 `Tuple` 参数。
**原因**：火山模型中，普通的 `Filter` 算子只有单个输入来源。而 `Join` 的谓词验证阶段，相当于在一个**联合的行结构**上进行求值。
**解决**：我们使用 `JoinedTuple` 作为代理视图模式，分别绑定对应的左行和右行数据，这样表达式在调用 `find_cell` 取值时，如果发现在左半边找不到字段，代理会自动去右半边查找，完美化解了多数据源上下文的问题。

### 3. Bison/Yacc 的语法歧义 (Shift/Reduce Conflicts)
**现象**：在 `yacc_sql.y` 中增加 `INNER JOIN` 的规约时，很容易和现有的查询 `WHERE` 子句甚至多表查询的 `From` 子句引发冲突。
**原因**：如果在表名列表 (`rel_list`) 和 `join_condition` 之间没有设定清晰的优先级，Bison 就无法确定是优先合并逗号，还是优先合并 JOIN 关键字。
**解决**：小心处理产生式层级，确保把 `INNER JOIN ... ON ...` 作为一个整体的 `RelListSqlNode` 节点，并且把它嵌套在原本的 `rel_list` 语法之上。这也让我深刻领会到编译原理中，规约路径对整个 AST 生成的影响。

## 📝 实验心得 (Reflections)

### 1. SQL 并没有“魔法”，只是朴素的逻辑组装
曾经我认为 SQL 引擎是非常神秘的黑盒，输入一句文本，就变魔术般查到了结果。但在实现完解析器并将其映射成 `LogicalOperator` (逻辑算子) 之后，我终于明白了它其实和普通的函数调用无异。**Parser 生成 AST -> Optimizer 改写 AST -> Physical Generator 选择算法 -> 执行算子树**。一条清晰的流水线将抽象的声明式语言转成了机器底层的控制流。

### 2. 火山模型 (Volcano Model) 的优雅与瓶颈
整个执行计划基于非常经典的火山模型：每个物理算子只需实现统一的 `open()`、`next()`、`close()` 接口。
- **优点**：这种设计极其优雅，算子与算子之间完美解耦，你可以像搭积木一样把 `Filter` 放在 `Join` 上，或者把 `Sort` 放在最顶端。
- **缺点**：为了多态调用，虚函数开销巨大，每次只处理一条记录也阻碍了 CPU 缓存和 SIMD 指令的发挥。这也是为什么现代分析型数据库 (OLAP) 更倾向于使用向量化执行 (Vectorized Execution) 的原因。

### 3. “算法课”在底层系统里的真实映射
在这个 Lab 里面，很多经典的算法开始有了实际意义：
- 为了避免 $O(N^2)$ 的嵌套循环代价，我们使用哈希表将连接的复杂度降低至 $O(M+N)$。
- 在 `PhysicalPlanGenerator` 中，我们通过检测 `ExprType::COMPARISON` 是否带有 `CompOp::EQUAL_TO` 的形式来决定是否能使用 `HashJoin`，这正是查询优化器中最简单的**基于规则的优化 (RBO)**。

这次完整的从顶到底的实现，让我对数据库在如何平衡抽象灵活性与底层执行效率上，有了更为具象的体感。

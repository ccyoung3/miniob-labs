# Lab 3 事务引擎与 WAL 持久化：并发隔离与状态重建的哲学

## 前言
数据库的核心能力不仅仅是快速地读写数据，更重要的是在并发环境中保证数据的一致性（ACID），并在面临突然断电等极端灾难时，能够利用日志准确无误地恢复出崩溃前的状态。在本次 Lab 3 的实验中，我基于 MiniOB 底层的 LSM-Tree 存储引擎，亲手搭建了多版本并发控制（MVCC）模型以及 Write-Ahead Logging (WAL) 持久化系统。

## 1. MVCC 快照隔离与事务合并读

### 设计思路
为了避免读写互相阻塞（特别是在高并发的线上环境），现代数据库多采用 MVCC。我们为每一个事务在开启时分配一个唯一的递增序列号 `seq_`（可以视作逻辑时间戳）。
- **写操作的缓存**：在事务执行期间，我们并未将未提交的数据直接刷入全局可见的 `MemTable`，而是用一个 `std::map<string, string>` 构成的 `inner_store_`（相当于 WriteBatch）将它们暂存起来。这就天然避免了“脏读”。
- **读操作的可见性**：事务内部在进行 `get` 查询时，优先查询 `inner_store_`，如果未命中，再带着当前的快照版本号去查全局存储。这样既保证了事务内能读到自己刚写的未提交数据，又不会读到别的事务在当前事务开启之后写入的新数据（这就实现了 Snapshot Isolation 快照隔离）。
- **范围查询与合并读**：最麻烦的地方在于 `TrxIterator` 的实现。如果要遍历所有数据，我们必须将 `inner_store_` 的本地更改与全局 LSM-Tree 迭代器通过多路归并（Merge）算法结合在一起。我构建了一个复杂的优先队列进行归并迭代，处理各种覆盖和被删除的标记。

### 踩坑记录：生命周期管理
一开始我直接将全局分配的事务管理器释放逻辑和事务对象的析构耦合在一起，导致内存泄漏和悬空指针。由于 `Db` 组件本身的层次较高，在跨越 `LsmTableEngine` 去销毁 `Trx` 对象时，我发现 `Db` 并没有直接提供对应的 `destroy_trx`，因为我们在 `open()` 的阶段实际上是利用了更底层的 `trx_kit()` 来主动创建 Vacuous（占位）事务。最后我通过深入分析 `Trx` 基类的虚析构行为，安全地采用了 `delete trx` 规避了内存泄露。

## 2. WAL 持久化与宕机恢复 (Recovery)

### 设计思路
在 LSM-Tree 中，新数据是直接写在内存（MemTable）中的，一旦宕机，内存中的数据就会全部丢失。为了保证 Durability，我们在写入内存前，必须先将操作写入到磁盘上的 WAL 文件中（Append-Only，速度极快）。
- **二进制序列化协议**：在这个实验里，我对数据格式进行了严苛的设计：
  - `[Sequence Number (uint64_t)]` + `[Key Length (size_t)]` + `[Key Bytes]` + `[Value Length (size_t)]` + `[Value Bytes]`。
- **故障恢复 (Recovery)**：服务器冷启动时，首先判断 WAL 文件是否存在。如果存在，则顺序按上面的二进制格式 `read_pos` 读取字节，反序列化成 `WalRecord`，再原样调用 `mem_table_->put()` 灌回内存中。这个过程实际上重建了宕机瞬间系统的完整状态。

### 踩坑记录：C++ 二进制 I/O
最开始在编写 WAL 的 `recover` 时，由于标准库并没有为我的工具类 `ObFileReader` 提供类似 `fread` 一样自动递增偏移量的底层 API（它只提供了基于绝对偏移量 `pos` 的 `read_pos`），我在解析二进制流时遇到了死循环，或者是读出的长度不匹配导致 Crash。
最后我严格维护了一个 `pos` 计数器，并使用 `sizeof(uint64_t)` 等精确字节数进行前移探测，并在每次访问前利用 `file_size` 严谨地做边界检查，避免越界读取引发 Segfault。

## 3. 自增主键 ID 的恢复
这是很容易被忽视但极其关键的一环。如果服务器重启，内存中的表引擎 `inc_id_`（负责自增的主键分配器）会变成 0。如果此时用户新插入一行数据，它分配的主键可能会和原来已经存在的数据的主键发生冲突！
为了解决这个问题，在 `LsmTableEngine::open()` 初始化期间，我构建了一个 `RecordScanner` 去遍历 LSM-Tree，利用我们系统内置的 `Codec::decode` 解码每一行数据的 `row_id`，并动态求取最大值。在此处，我不得不稍微修改了系统的 `Codec::decode` 接口，增加了能一次性解析出 `table_id` 和 `rid` 的方法，完美地初始化了自增键的起点。

## 总结
通过本次实验，我才真正理解了为什么说“数据库其实就是一个巨大的状态机”。从 WAL 重播到 MemTable，从多版本快照到合并读取，每一步设计都在处理一致性、持久化与性能三者之间的 Trade-off。这不仅是工程能力的锻炼，更是一次对软件架构设计的深度思考。

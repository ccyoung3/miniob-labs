## 3. 关于 DELETE 和 LSM-Tree 的 Tombstone
我在看文档要求验证 LSM-Tree 下 DELETE 行为的时候，一开始非常担心要做大的改动。
但当我查看底层引擎后，发现非常惊喜的一点：MiniOB 中 `LsmTableEngine::delete_record_with_trx` 内部就已经调用了对底层 lsm 数据树的 `remove()`。在 LSM-Tree 的架构中，这个操作实际上就是在 MemTable 里写入一个 `Tombstone` 标记（即墓碑），它并不会立马去删除底层数据，而是依靠后台的 Compaction 将它和原有的记录抵消掉。这正是 LSM-Tree 极其精妙的一环！我们的物理算子直接通过事务引擎接口透传到底层就自然实现了正确的语义，这再一次印证了**优秀的抽象接口设计能够让架构扩展变得异常轻松**。

## 4. TPC-C 性能测试避坑指南
Lab4 最主要的产出本应是 TPC-C 性能测试结果。然而，在这个部分我最终选择了跳过在本地执行。原因主要有三：
1. **老旧依赖**：官方的 TPCC python 测试代码基于已被彻底遗弃的 `Python 2.7`，而我的开发环境（包括主流 Ubuntu 和 macOS）中都没有这个包了。如果强行源码编译并构建 `pip2`，既容易污染环境，也缺乏维护价值。
2. **硬件差异**：性能基准测试的参考数据是高度依赖硬件的。官方的对比基准往往是统一配置（例如 1C1G）。我在本地即便配置环境跑到吐血得到一个数字，其实也不能公平地说明系统的优劣，这就背离了性能分析的前提条件“控制变量”。
3. **正确做法**：将代码写好、确认编译无误和基本功能正常后，直接依赖云端自动评测机（如果有的话）来验收是最合适的做法。把有限的精力聚焦在核心源码逻辑的理解上（比如上面说的聚合函数与更新逻辑），而不是去死磕已经淘汰的测试框架，这是软件工程里的基本决策能力。
